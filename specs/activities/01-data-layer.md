## 1. Database Migration

**File:** `supabase/migrations/202603080001_add_activities.sql`

### Schema

```sql
create type activity_category as enum ('meal', 'tour', 'ticketed', 'general');

create table activities (
  activity_id bigint generated by default as identity primary key,
  destination_id bigint not null references destinations (destination_id) on delete cascade,
  category activity_category not null default 'general',
  name text not null,
  day_number integer not null check (day_number >= 1),
  start_time time,
  end_time time,
  position integer not null default 0 check (position >= 0),
  notes text,
  details jsonb not null default '{}',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

### Required ancillary objects

1. **Index** for the primary query pattern (all activities for a destination, sorted by day + position):
   ```sql
   create index idx_activities_destination_day
     on activities (destination_id, day_number, position);
   ```

2. **Trigger** reusing the existing `update_updated_at()` function (defined in `202602030001_create_schema.sql`):
   ```sql
   drop trigger if exists activities_updated_at on activities;
   create trigger activities_updated_at
   before update on activities
   for each row execute function update_updated_at();
   ```

3. **RLS** -- same join pattern as accommodations (destination -> trip -> user):
   ```sql
   alter table activities enable row level security;
   alter table activities force row level security;

   drop policy if exists "Users manage own activities" on activities;
   create policy "Users manage own activities"
     on activities
     for all
     to authenticated
     using (
       exists (
         select 1
         from destinations as d
         join trips as t on t.trip_id = d.trip_id
         where d.destination_id = activities.destination_id
           and t.user_id = (select current_user_id())
       )
     )
     with check (
       exists (
         select 1
         from destinations as d
         join trips as t on t.trip_id = d.trip_id
         where d.destination_id = activities.destination_id
           and t.user_id = (select current_user_id())
       )
     );
   ```

4. **Grants** -- the default privileges in `202602030001` cover new tables for `authenticated`. No extra grants needed unless the enum type needs explicit usage grants (verify during migration testing).

### Implementation suggestions

> These are suggestions, not prescriptions. Adapt to what makes sense during implementation.

- Wrap the enum creation in a `DO $$ ... $$` block with an existence check, matching the pattern in `202602030001_create_schema.sql` for `transport_type` and `transport_role`.
- Consider whether `check (day_number >= 1)` is correct for your data model. Day 1 = first day at destination. A stopover with `duration = 0` should not show the schedule button, so `day_number >= 1` is safe.

## 2. TypeScript Types

**File:** `src/types/database.ts`

### New types to add

```typescript
export type ActivityCategory = 'meal' | 'tour' | 'ticketed' | 'general';

export interface Activity {
  activity_id: number;
  destination_id: number;
  category: ActivityCategory;
  name: string;
  day_number: number;
  start_time: string | null;
  end_time: string | null;
  position: number;
  notes: string | null;
  details: Record<string, unknown>;
  created_at: string;
  updated_at: string;
}
```

### Existing type to modify

Add `activities` to `DestinationWithRelations`:

```typescript
export interface DestinationWithRelations extends Destination {
  transport: Transport | null;
  accommodation: Accommodation | null;
  activities: Activity[];  // <-- new
}
```

### Impact of this change

Adding `activities: Activity[]` to `DestinationWithRelations` will cause type errors in every place that constructs this type manually. These places need `activities: []` added:

- `destination-list.tsx` -- in the `ADD_ITEM` case of the reducer (`{ ...action.destination, transport: null, accommodation: null }`)
- `destinations.ts` server action -- `saveDestinationDetailsAction` return value
- `trips.ts` query -- `getTripById` where it maps destinations
- `import-export.ts` -- if it constructs `DestinationWithRelations` anywhere (it doesn't currently, but verify)

### Implementation suggestions

> These are suggestions. The `details` field typing could be more specific (e.g., discriminated union per category), but `Record<string, unknown>` keeps it simple and matches the JSONB column. Consider adding category-specific types later if form validation needs them.

## 3. Database Queries

**File:** `src/lib/db/queries/activities.ts` (new)

Follow the pattern from `src/lib/db/queries/accommodations.ts`.

### Functions to implement

#### `getActivitiesByDestination(destinationId: number): Promise<Activity[]>`
- Validate `destinationId` with `isValidId`
- Query `activities` table filtered by `destination_id`
- Order by `day_number asc, position asc`
- Return empty array if none found

#### `createActivity(input): Promise<Activity>`
Input shape:
```typescript
type ActivityCreateInput = {
  destination_id: number;
  category: ActivityCategory;
  name: string;
  day_number: number;
  start_time?: string | null;
  end_time?: string | null;
  position?: number;
  notes?: string | null;
  details?: Record<string, unknown>;
};
```
- Validate `destination_id` and `day_number`
- Trim and normalize `name` (reject if empty after trim)
- Normalize optional text fields with `normalizeOptionalText`
- If `position` not provided, compute it: count existing activities for that destination+day, use count as position
- Insert and return with `.select('*').single()`

#### `updateActivity(activityId: number, updates): Promise<Activity>`
Input shape:
```typescript
type ActivityUpdateInput = {
  category?: ActivityCategory;
  name?: string;
  day_number?: number;
  start_time?: string | null;
  end_time?: string | null;
  position?: number;
  notes?: string | null;
  details?: Record<string, unknown>;
};
```
- Validate `activityId`
- Build update object from provided fields (skip undefined)
- If `name` provided, trim and validate non-empty
- Update and return

#### `deleteActivity(activityId: number): Promise<void>`
- Validate `activityId`
- Delete by `activity_id`

### Implementation suggestions

> Consider reusing the `normalizeOptionalText` and `isValidId` helpers from accommodations.ts. You could extract them to a shared utils file, or just duplicate them -- the duplication is minimal and avoids coupling. The codebase currently duplicates them, so follow that pattern.
